API Endpoints & Database Schema

Complete finalized reference for all four microservices. Covers every controller endpoint, HTTP method, access role, request/response notes, and the full postgres schema per service database.

Access roles: PUBLIC — no token required CUSTOMER — JWT with ROLE_CUSTOMER ADMIN — JWT with ROLE_ADMIN All protected routes validated at Gateway via JwtAuthFilter
🔐

Auth Service

Port 8081 · /api/auth/** · AuthController.java
JWT issued via JJWT. Token payload includes userId, email, role, iat, exp. Expiry: 24h access token. Refresh token stored in DB. Password hashed with BCrypt (strength 12).
AuthController Authentication, registration, token management
Method Path Description Access
POST /api/auth/signup Register new customer account
Body: name, email, mobile, password → returns AuthResponse (token + profile)
 PUBLIC
POST /api/auth/login Login with email + password
Returns accessToken, refreshToken, expiresIn, userRole
 PUBLIC
POST /api/auth/refresh Exchange refresh token for new access token
Body: { refreshToken }. Validates against DB before issuing.
 PUBLIC
POST /api/auth/logout Invalidate refresh token
Deletes refresh token record from DB. Access token expires naturally.
 CUST + ADMIN
GET /api/auth/me Get own profile from token
Returns UserProfileResponse (id, name, email, mobile, role, status, createdAt)
 CUST + ADMIN
PUT /api/auth/me Update own profile (name, mobile)
Email and role changes not permitted here
 CUSTOMER
POST /api/auth/change-password Change password with old password verification
Body: { currentPassword, newPassword }
 CUST + ADMIN
POST /api/auth/forgot-password Initiate password reset (OTP/link via email)
Generates a reset token stored in password_reset_tokens table
 PUBLIC
POST /api/auth/reset-password Complete password reset with token
Body: { token, newPassword }. Token expires in 15 min.
 PUBLIC
GET /api/auth/validate Internal-only: validate JWT and return claims
Used by Gateway JwtAuthFilter. Not exposed externally.
 INTERNAL
GET /api/auth/admin/users List all users with filters (role, status, search)
Paginated. Params: page, size, role, status, q
 ADMIN
GET /api/auth/admin/users/{id} Get any user's profile by ID ADMIN
PATCH /api/auth/admin/users/{id}/status Activate / deactivate a user account
Body: { status: ACTIVE | INACTIVE | SUSPENDED }
 ADMIN
POST /api/auth/admin/users Create an admin-level user
Only ADMIN can set role=ADMIN during creation
 ADMIN

Gateway Routing Rules

GatewayConfig.java Spring Cloud Gateway route predicates + filters
Route Pattern Forwards To JWT Required Role Check
/api/auth/signup, /api/auth/login, /api/auth/refresh, /api/auth/forgot-password, /api/auth/reset-password auth-service:8081 No None
/api/auth/** auth-service:8081 Yes CUSTOMER or ADMIN
/api/catalog/medicines (GET), /api/catalog/categories (GET) catalog-service:8082 No None
/api/catalog/** catalog-service:8082 Yes CUSTOMER or ADMIN
/api/orders/** order-service:8083 Yes CUSTOMER or ADMIN
/api/admin/** admin-service:8084 Yes ADMIN only
The Gateway extracts userId and role from the JWT and forwards them as X-User-Id and X-User-Role headers to downstream services. Services read these headers instead of re-validating the token, which keeps each service stateless. The /api/orders/payments/callback webhook is whitelisted at the Gateway to bypass JWT (it uses HMAC signature verification instead).
